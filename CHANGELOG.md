## [2026-07-01] UI: card actions collapsed into a single "⋯" menu

The per-card action bar (Save / Share / **Not for me**, the dismiss pushed bottom-right) was a big one-tap target sitting in the scroll path — easy to misclick into "not interested" while scrolling — and three buttons on every card was noisy. Replaced it with a single small **"⋯"** button at the card's **bottom-left** (off the right-side scroll path) that opens a compact popover menu: **Save · Share · Not interested**. Now a stray tap just opens a dismissable menu instead of instantly hiding a story (dismiss is a deliberate two-tap), and the feed is much quieter (one tiny icon vs. a 3-button bar). Save shows a "Saved" toast + reflects state on open; the "Not interested" Undo flow is unchanged; menus close on outside-click/scroll. Applies everywhere rows render (feed, niche pages, Catch-up, Search). `index.html` only — `actionBar()`/`handleAction()` replaced by `cardMenu()`/`cardAction()`/`toggleMenu()`.

## [2026-07-01] Feat: topic-balanced Catch-up brief (engine `catchup` section)

Fixes the alpha cohort's "Catch-up isn't balanced or niche-aware" finding. The old UI `renderCatchup()` was `visiblePicks().filter(score>=8).slice(0,8)` — top-8 by score, which on real data was a **wall of sports** (measured: 5 Sports + 3 Politics). New **engine-side** `catchup` section in `sections.py`: a **topic-balanced, bounded digest** (`CATCHUP_LIMIT = 12`, `CATCHUP_PER_CATEGORY = 3`) built by **round-robining across niches** in feed-rank order — the biggest story from the biggest-story niche, then the next niche, etc. Measured on the same live feed: **one item per niche across 12 niches** (LeBron, Supreme Court, fusion energy, markets, Xbox, SCOTUS location-data, Meta, Mazda, Madonna, climate, RFK/FDA, Bitcoin) — high-signal (leads with the score-10s) and **already deduped** (built from the cluster-collapsed flat feed, so LeBron shows "+1 more" not two cards). Pure data, **no LLM/$0**. The engine can't know a user's picked niches or last-open time (no accounts), so it emits a balanced **superset**; the **UI** (its lane) filters to picked niches (`blank.niches`) + windows to "since you last opened" (`lastOpen`) + adds the "you're all caught up" closure beat. Engine/UI split per the ownership boundary — `sections.py` only, `index.html` untouched. `FEED_CONTRACT.md` updated with the `catchup` rail spec + the now-live clustering fields (`cluster_size`/`cluster_sources`/`related_articles`) for the UI agent. The on-tap "catch me up" **summary** (the one paid, per-user LLM feature) remains deferred to Phase 3.

## [2026-07-01] Ops: ease engine cadence 10 → ~15 min (off-hour offset) vs GitHub throttling

`blank.yml` cron changed `*/10 * * * *` → `7,22,37,52 * * * *` (~every 15 min, offset off the hour). **Why:** GitHub's `schedule:` trigger is best-effort and **deprioritizes the highest-frequency workflows first under load** — the every-10-min engine was getting silently dropped for hours (06-30: stalled ~3.5h, feed froze on dated items). A less-frequent schedule is honored more reliably, so a nominal 15-min cadence yields *more consistent* real runs than an unreliable 10-min; the `7,22,37,52` offset avoids GitHub's **top-of-hour peak-load window** (they explicitly recommend not scheduling on the hour). Bonus: fewer runs = fewer per-run Moment calls (small cost trim). **This is a mitigation, not a cure** — it reduces the odds of being dropped but still rides GitHub's best-effort scheduler; the durable fix (still open) is an external `workflow_dispatch` trigger (a free cron service hitting the API), which bypasses scheduled-workflow deprioritization entirely. Feed freshness drops 10→15 min nominal (imperceptible to readers).

## [2026-07-01] Multi-source story clustering (dedup) — deterministic, zero-cost

Fixes the 10-persona alpha cohort's **#1 finding**: the feed served the same event 6–11× as separate cards (LeBron, Microsoft layoffs, etc.) because the engine only did exact `content_hash` dedup — the `clusterBlock()` UI drawer was built but the payload always shipped `cluster_size:1`/`cluster_sources:null`/`related_articles:null`. New **`cluster.py`** stage groups same-event items across sources and the feed builder collapses them into one card. **Approach = deterministic, no LLM/API (founder constraint: zero added cost).** Each candidate window (scored ≥6, 48h) is clustered by a **strong-entity anchor** (proper nouns, excluding months/days/generic caps — the fix that keeps "Nissan sales fall in May" and "Toyota sales fall in May" apart) **AND** a title word-Jaccard gate (`WORD_JACCARD_THRESHOLD = 0.42`, tuned on real `blank.db` data), unioned via union-find. Runs in ~0.4s over ~1,200 items; deterministic → cluster identity (`content_key` = lead's `content_hash`) is stable across runs with no bookkeeping. Lead = highest score, tiebreak toward an item **with an image** (also fixes gray-hero cards), never a rewritten headline. New `clusters` table + `db.replace_clusters()`/`get_active_clusters()`; stage wired into `run_pipeline.py` after Moment. `deploy-pages.yml` now **collapses**: ranks each cluster once by its lead, drops members from the flat stream, and fills `cluster_size`/`cluster_sources`/`related_articles` so the existing "Covered by ESPN, CBS +N" drawer lights up (graceful — any failure leaves the feed uncollapsed, exactly today's behavior). Measured on live data: 48 clusters, **64 duplicate cards folded** (Gene Wilder AI ×5, Ja Morant ×4, Supreme Court birthright unified ×4, Rocket Lab ×3, Microsoft, LeBron…), **0 wrong merges** in audit. Plan: `brainstorms/2026-07-01-clustering-plan.md`. `moment.py` intentionally left as-is (semantic umbrella events are a different granularity than same-story dedup). Engine-owned; `index.html` untouched.

## [2026-06-30] Fix: recover real images from bot-walled publishers (ESPN et al.) via crawler UA

The Top Stories hero was rendering gray — its cards were **ESPN**, which (like Billboard, Rolling Stone) sits behind a **bot wall**: a normal browser User-Agent gets **HTTP 202 + empty body**, so OG enrichment found no image and ESPN's RSS carries none inline. Verified empirically that these publishers **whitelist crawlers** (they want link-preview + search indexing): the same ESPN URL returns **200 + the real `og:image`** to `facebookexternalhit` / Googlebot / Bingbot. Fix: `ingest._fetch_og_image()` now tries a **social-crawler UA (`facebookexternalhit`) first** — the exact purpose og:image exists for — then falls back to the browser UA (image parsing factored into `_extract_image()`). This pulls the **actual article photo** (`a4.espncdn.com/…`), no placeholders/stock. Verified end-to-end: ESPN hero + FAQ + a TechCrunch article all now return their real images. Also bumped `OG_MAX_ATTEMPTS` 3 → 5 so items that already failed under the old browser-only UA get retried with the crawler UA (recovers the currently-visible imageless backlog over the next few enrich passes). Complementary UI task (flagged to the UI agent): the hero/prominent cards should prefer the imaged version of a story when multiple sources cover it.

## [2026-06-30] Tune: feed time-decay 12h → 6h per point (fresher feed)

`DECAY_HOURS_PER_POINT` in `db.py` (the single source of truth for feed ordering, shared by `db.get_feed()` and the `deploy-pages.yml` builder) lowered **12 → 6** so recency competes harder with score. At 12h/point a strong story (score 9) pinned the top for ~36h, so the feed always led with several-hours-old "dated" items and fresh news couldn't break in. At 6h/point: `rank = score − ageHours/6`, so a fresh score-8 (8.0) now beats a 7h-old score-9 (7.83), and a score-9 leads ~18h instead of ~36h before a fresh 6 passes it — big stories still lead while genuinely fresh, but the feed stops feeling stale. One constant, easily re-tuned. (Note: this only takes effect while the engine + deploy are actually running — see the separate engine-scheduling issue.)

## [2026-06-30] Fix: engine robustness from code review (RSS timeout, OG URLs, DB handles)

Three verified fixes from a multi-agent code review (the reviewer's "HIGH item-index mismatch" flags were false positives — the triage/score id-mapping is well-defended — and were correctly left alone). **(1) RSS fetch had no network timeout** (`ingest.py` `poll_source`): `feedparser.parse(url)` does a blocking urllib fetch with no timeout, so a single hung/slow source could stall the entire serial, single-concurrency pipeline (and queue the next cron behind it) — the likely cause of "Reddit timing out." Now fetches bytes via `requests.get(url, timeout=15)` and hands them to `feedparser.parse(resp.content)` (encoding still auto-detected from raw bytes), with the HTTP-status/error handling preserved; `trend://` pseudo-sources are skipped early (they're populated by `fetch_trends`, not polled). **(2) OG-image scraping dropped valid images** (`ingest.py` `_fetch_og_image`): root-relative URLs (`/share.jpg`) were discarded and `&amp;` in CDN URLs wasn't unescaped (→ 404s). Now `html.unescape()`s the URL and `urljoin()`s relative paths against the final (post-redirect) article URL — recovers images that were silently lost. **(3) DB connections were never closed** (`db.py` `_conn`): the helper returned a bare connection and callers' `with` only commits/rolls back, leaking handles (worst on Windows, where a lingering handle blocks replacing `blank.db`). `_conn` is now a `@contextmanager` that preserves commit-on-success/rollback-on-error via an inner `with con:` and closes in `finally` — a drop-in (all 17 callers already use `with _conn(...)`). Verified: all module self-tests pass; a live poll of The Verge ingested 10 items through the new path. Deliberately skipped: the dead `publish.py` XSS (nothing imports it — the live renderer is `index.html`), undated-item ranking (frozen at ingest, not "pinned forever"), and the score-omission/recency-window items (load-bearing — better as deliberate tasks).

## [2026-06-29] Fix: scoring stalled — raise Sonnet max_tokens (truncated JSON)

New stories stopped reaching the feed because the **Score stage was writing ~0 items per run**. Root cause: `score.py` had `max_tokens=1024` while each item's JSON output now includes `primary_category` + up to 5 `tags` (added Jun 28) on top of score + 4 criteria. A 12-item batch emits ~1,100+ tokens, overflowing the 1,024 ceiling, so Sonnet's reply was truncated mid-array (`"tags": ["lebron ja`…) → `_parse_sonnet_response()` returned `None` → the entire batch was skipped. The latest run showed `Scored: 0, Skipped: 420` with all 35 batches failing to parse; scores/hour had collapsed from 50–80 (Jun 28) to 1–9 (Jun 29), and the feed (built only from scored items) went stale. Fix: raised `max_tokens` 1024 → 4096 (a ceiling, not a target — the model emits only what it needs, ~1.0–1.3k/batch, so steady-state cost is unchanged; it just stops truncating). Also added `message.stop_reason` to the parse-failure log line so a future `max_tokens` truncation is diagnosable at a glance instead of mistaken for malformed JSON. The ~1,500-item escalated backlog drains over the next several runs (bounded by `PER_SOURCE_CAP`). Ingest/triage/OG-enrich/backfill/moment were all healthy; only scoring was affected.

## [2026-06-29] Feat: self-draining topic-tag backfill

Topic tagging (`primary_category`/`tags`) is produced in the Sonnet score pass, so only items scored *after* it shipped carry tags — leaving the category rails nearly empty (~9 tagged of ~1,476 recent scores). New `backfill_tags.py` adds a self-draining engine stage that tags the leftover backlog: each run it pulls up to `BACKFILL_LIMIT = 40` scored, feed-eligible (`score>=6`, last 48h), still-untagged items (newest-first) via one cheap **Haiku** call per batch of 20, and writes `primary_category`/`tags` back through the new `db.update_score_tags()` (topic fields only — it never re-scores). Candidates come from the new `db.get_untagged_scored_items()`. It reuses `score.py`'s taxonomy (`CATEGORIES`), item formatting, JSON parser, and normalizers so backfilled tags match the score pass. **Self-draining:** new scores are already tagged and old untagged items age out of the 48h window, so the queue empties within a few runs (~613 backlog → ~40/run) and the stage becomes a near-free no-op (one COUNT query). Wired into `run_pipeline.py` after Score (stage 3.5); graceful no-op without `ANTHROPIC_API_KEY`. Runs in CI so `blank.db` writes commit through the normal engine flow. Cost stays global-score-once: bounded cheap Haiku classification, never per-user.

## [2026-06-29] Feat: sectioned feed — Top Stories, Moment, and Category rails

Moves the feed from a single flat ranked stream toward an Apple-News-style **sectioned** page, building the **engine + data** for three v1 rails (UI rendering is a separate effort; this only produces the JSON contract). **(1) Moment detection** — a new `moment.py` engine stage (one **Haiku** call per run, mirroring `triage.py`) identifies "the single biggest thing the world is collectively following right now," over ~60 recent scored headlines. It deliberately **allows sustained mega-events** (a World Cup, the Olympics, an election) — not just single breaking events — and chooses by **breadth of coverage** (the event the most stories are about), so the candidate pool is ordered by recency/volume rather than score (a handful of high-scored breaking items must not bury a high-volume mega-event). It's a deliberately **narrow slice of clustering** (find THE one event, not full multi-source clustering, which stays parked), sourced from scored news so it can't go stale like the old trends surface. Members are score-ordered (rail lead = strongest story) and capped at 12. Requires ≥3 member stories or it reports no moment (graceful absence); no API key / API failure / parse failure all no-op cleanly. Stored in a new `moments` table in `blank.db` (`db.record_moment()` / `get_active_moment()` / `clear_moment()`; only one row active at a time). Wired into `run_pipeline.py` after Score. **(2) Section assembly** — a new importable `sections.py` (`build_sections()`, pure data, no LLM) turns the already-ranked flat picks + the active moment into ordered rails: `moment` (lead + supporting; its items pulled OUT of Top Stories so the page never repeats), `top_stories` (hero = `items[0]`, up to 6), and `category` rails grouped by `primary_category` (the score-pass tag), ordered by size, ≥2 items each, excluding empty/`Other`. **(3) Payload** — `deploy-pages.yml` now selects `i.id` + `s.primary_category`, carries `item_id`/`primary_category` on each pick, and emits a new top-level **`sections`** key — **additive**: the flat `runs` stream is untouched and remains the fallback, so the current PWA keeps working until the UI consumes rails. Assembly is wrapped so any failure (e.g. a deploy before the engine's next migration → no `moments` table) just omits `sections`. Category rails are non-personalized for now (niche onboarding will filter them per-user) and will be sparse until topic-tagging coverage grows. Contract documented in new `FEED_CONTRACT.md`. Cost rule preserved: one cheap global Haiku call per run, no per-user LLM work, no new vendor.

## [2026-06-28] Feat: topic tagging in the score pass (Phase 2 seed)

Sonnet now classifies every item *during the existing global score call* — no extra LLM pass, just a few more output tokens/item — producing a fixed `primary_category` plus free-form `tags`. This is the hook for per-user niche matching later (niche onboarding maps to `primary_category`; `tags` give discovery/search granularity). The shape is **both**: one of 15 curated buckets (Politics & Policy, World / International, Business & Economy, Technology & AI, Science, Health & Medicine, Climate & Environment, Sports, Entertainment, Gaming, Arts Books & Ideas, Internet Culture, Lifestyle, Crime & Justice, Money & Personal Finance) + an `Other` escape hatch so nothing is force-fit. `score.CATEGORIES` is the single source of truth and is injected into the scoring prompt. `score._normalize_category()` snaps the model's label to a canonical bucket case-insensitively (anything unrecognized → `Other`), and `_normalize_tags()` trims/caps tags (≤5, ≤50 chars each, drops empties/non-strings) so drift never corrupts the taxonomy. Storage: two new columns on the `scores` table (`primary_category TEXT`, `tags TEXT` JSON), added via idempotent `init_db()` migrations so older `blank.db` files migrate cleanly on the next engine run; `db.record_score()` and `db.get_feed()` carry both fields. `deploy-pages.yml` is intentionally untouched (it selects only the columns it needs), so the live feed is unaffected until the PWA consumes niches. Global-scoring cost rule preserved: still one feed scored once.

## [2026-06-27] UI: feed polish — living chrome, rarity gems, graceful images

A round of refinements to the redesigned feed in `index.html`, all aimed at making it feel more "alive" and premium. **Chrome that gets out of the way:** the top bar auto-hides on scroll-down and slides back on scroll-up (always shown at the very top). **Rubber-band pull-to-refresh:** the whole feed now drags down with the finger (with resistance), holds open while refreshing with the spinner in the revealed gap, then snaps back — instead of just growing a detached spinner. **Rarity gems:** the score signal is now a single small colored diamond keyed to score (blue = 8 Rare, purple = 9 Epic, gold = 10 Legendary; 6–7 unmarked), replacing the interim "Editor's Pick"/"Notable" labels and the green card-border glow (which, because ~29% of items score ≥8 and cluster at the top, read as ambient green rather than a signal). **Graceful images:** rows without an image render as clean full-width text (no empty tile), with headlines line-clamped so heights stay even; the hero verifies image resolution before rendering and walks down the ranked stories to lead with the highest-ranked one that has a genuinely good-res photo, rejecting tiny/low-res images (e.g. 140px Reddit thumbs, which are signed URLs and can't be upscaled) with a texture fallback; image crops are biased toward the top so faces aren't sliced off.

## [2026-06-27] Refactor: move time-decayed feed ranking into the engine (one source of truth)

The "ranked living feed" ordering (rank = score − ageHours / `DECAY_HOURS_PER_POINT`, recency as tiebreak) was computed client-side in `index.html`, so surfaces could drift. Centralized it: `DECAY_HOURS_PER_POINT = 12` plus `decayed_rank()` / `rank_timestamp()` now live in `db.py` as the single tunable source of truth. `db.get_feed()` orders its candidate pool by decayed rank before source-balancing. `deploy-pages.yml` imports `db` and — after assembling all runs — flattens, dedups, and sorts every pick by decayed rank, emitting one ranked run so `picks_data.json` arrives already in final order. `index.html`'s client-side sort and its `DECAY_HOURS_PER_POINT` / `rankScore` / `ageHours` / `itemDate` helpers were removed; the page now renders the order it's given. Age basis is `published_at`, falling back to `scored_at` / run time, with no date → age 0 (graceful). The 48h retention window is unchanged.
